## 2. 总体结论

总体判断：

- 当前这批 `test` 数据在“episode 结构、path 归属、prefix state/graph、tree backup 标签”这些核心语义上，整体是对的。

## 5. 风险项排查结果

## R1. `step_data`/`reasoning_results`/`tool_results` 是硬截断，不是语义摘要

- 排查目的：
  - 判断 state 中供模型使用的文本证据是否发生严重信息压缩
- 潜在风险：
  - 当前 world model 训练并不读取原始 agent 输出，而是读取截断后的 summary
  - 如果截断发生在关键结论、后置证据、反驳信息之前，模型会学到不完整甚至误导性的 state representation
  - 对 search/tool 步尤其严重，因为真正有用的信息常常出现在较后位置，而不是前 240/300 字
- 代码位置：
  - `puppeteer/inference/policy/REINFORCE_continuous.py:521-527`
  - `puppeteer/inference/policy/workflow_dataset_recorder.py:723-729`
  - `puppeteer/inference/policy/workflow_dataset_recorder.py:422-490`
- 排查结果：
  - `_summarize_text()` 实际是简单的 `text[:limit] + "..."`，不是 LLM 摘要
  - 在本批样本中：
    - 75 个 action 里，51 个原始 `step_data` 长度超过 240
    - 这 51 个 action 的 `metadata.result_summary` 全部发生截断
    - 这 51 个 action 的 `state.executed_steps[].step_data_summary` 也全部发生截断
  - 长度统计：
    - 原始 `step_data` 平均长度约 `1932`
    - `metadata.result_summary` 平均长度约 `239`
    - `state.executed_steps[].step_data_summary` 平均长度约 `293`
  - 具体样例：
    - 某个 Tavily 搜索结果原始长度约 `5000`，最终只保留前 `243` 字
    - 某个 reasoning 结果原始长度约 `2328`，最终只保留前 `243` 字
- 影响判断：
  - 高风险
  - 其中 `metadata.result_summary` 当前 adapter 不直接使用，影响较小
  - 但 `state.executed_steps[].step_data_summary`、`reasoning_results`、`tool_results` 当前 adapter 会直接读取，影响是实质性的
- 建议人工再排查：
  - 随机抽 20 条长 reasoning/search step，对比“原文最后 30%”里是否经常出现关键证据
  - 重点看：
    - 最终结论是否常常在被截掉的尾部
    - 搜索结果中的真正相关 snippet 是否常常不在前 300 字
  - 如果结论是“尾部经常有关键内容”，应改成：
    - 结构化摘要
    - 或提取 top-k 证据句
    - 或保留更长窗口并区分 search/reasoning 两类文本

## R2. `valid_action_mask` 在本批数据中是常量标签

- 排查目的：
  - 判断 valid-action 头是否真的在学习“当前状态下哪些 action 有效”
- 潜在风险：
  - 如果标签恒定，模型会把 valid head 学成常量分类器
  - 指标可能很好看，但几乎没有调度价值
- 代码位置：
  - `puppeteer/inference/policy/workflow_dataset_recorder.py:537`
  - `puppeteer/inference/policy/workflow_world_model.py:1159`
- 排查结果：
  - 本批 75 条 record 的 `next_state_targets.valid_action_mask` 只有 1 个唯一值
  - 也就是 75/75 都是同一份 agent 列表
  - 该列表中还包含 4 个当前数据里从未真正被执行过的 agent：
    - `Modifier_gpt4o`
    - `PythonAgent_gpt4o`
    - `ReflectAgent_gpt4o`
    - `SummarizerAgent_gpt4o`
  - 与之相对，`state.workflow_valid_actions` 在本批有 25 种不同取值，反而更有信息量
- 影响判断：
  - 高风险
  - 这会直接影响 valid-action 头训练价值
- 建议人工再排查：
  - 明确 valid head 的目标到底应是：
    - “当前可调度的 agent”
    - 还是“当前前缀下已验证有效的 action 类型”
  - 如果目标是 scheduler 级别调度，建议不要再用全量 agent 列表当 gold
  - 建议改为：
    - scheduler 当步候选集
    - 或基于规则/回溯定义的真实可行动作集合

## R3. action 词表把“agent 名”和“动作名”混在了一起

- 排查目的：
  - 判断 world model 的 action-related embedding 是否语义一致
- 潜在风险：
  - 当前模型在同一个 `action_to_id` 词表里同时编码：
    - 当前动作：agent 名，例如 `ReasoningAgent_gpt4o`
    - 历史 step：动作名，例如 `reasoning`、`search_tavily`
    - valid-action 标签：又回到 agent 名
  - 这会导致：
    - `step_action_ids`、`action_name_ids`、`next_valid_mask` 其实在共享一个混合语义空间
    - valid head 学的是 agent 空间，而历史轨迹用的是 verb 空间
- 代码位置：
  - `puppeteer/inference/policy/workflow_world_model.py:1062`
  - `puppeteer/inference/policy/workflow_world_model.py:1135`
  - `puppeteer/inference/policy/workflow_world_model.py:1159`
- 排查结果：
  - 本批数据中：
    - 当前动作名集合是 agent roles，例如 `TavilyAgent`、`ReasoningAgent_gpt4o`
    - 历史 step 动作集合是 verbs，例如 `reasoning`、`search_tavily`、`conclude`
    - 两者交集为空
  - 说明这不是“同一概念的两种拼写”，而是两套完全不同的语义空间
- 影响判断：
  - 高风险
  - 这不是单纯的文档问题，而是当前训练消费层面的结构问题
- 建议人工再排查：
  - 明确模型到底想预测：
    - agent selection
    - 还是 atomic action verb
  - 建议把词表拆开：
    - `agent_to_id`
    - `tool_or_action_to_id`
    - `valid_agent_mask`
  - 如果不拆，至少要统一 `current action` 与 `history action` 的语义层级

## R4. scheduler 存在 JSON 解析失败并回退到 role-name matching

- 排查目的：
  - 判断采样路径是否严格来自 scheduler 的结构化 JSON 决策
- 潜在风险：
  - 如果 scheduler 没有输出可解析 JSON，而系统改用“从原文中匹配 agent 名”的回退逻辑：
    - 采样的 path 会带额外噪声
    - confidence 会被拉平成默认值
    - 选出来的 agent 可能并不是模型最终 intended JSON
  - world model 学到的将是“带 fallback 噪声的采样策略”而不是纯 scheduler policy
- 证据位置：
  - `3-2026-04-02-12-06-14/train.log:7201`
  - `5-2026-04-02-13-30-35/train.log:12632`
  - `7-2026-04-02-14-19-50/train.log:16938`
  - `7-2026-04-02-14-19-50/train.log:31827`
  - `7-2026-04-02-14-19-50/train.log:46116`
- 排查结果：
  - 本 session 至少出现了 5 次：
    - `Scheduler JSON parsing failed, falling back to multi-agent role-name matching.`
  - fallback 后日志里出现了典型的均匀置信度结果，例如：
    - `raw_confidence: 0.25`
    - `sharpened_confidence: 0.25`
  - 受影响 run 包括：
    - run 3
    - run 5
    - run 7（出现 3 次）
- 影响判断：
  - 高风险
  - 它影响的是采样路径本身，不只是某个无关日志字段
- 建议人工再排查：
  - 随机抽取这 5 次 fallback 前后的 scheduler 原始回复，确认 fallback 选出来的 agent 是否真的等价于 intended JSON
  - 如果不等价，应优先修 parser 或强制模型返回 JSON
  - 如果 fallback 不可避免，建议把“是否 fallback”单独打进 dataset metadata

## R5. later-step state 中经常已经出现答案字母，容易形成 reward/value shortcut

- 排查目的：
  - 判断模型是否会通过“已经暴露出的答案”来轻松预测 reward/value，而不是学习真正的过程动态
- 潜在风险：
  - 对后期 transition，reward/value 可能不再需要通过 reasoning/evidence 结构判断，只要看到答案字母即可
  - 这会使模型更擅长“评估后期状态”，但不一定更擅长“从早期状态规划”
- 相关代码：
  - `puppeteer/inference/policy/workflow_dataset_recorder.py:406-490`
  - `puppeteer/inference/policy/workflow_world_model.py:1081`
- 排查结果：
  - 本批 75 条 record 中：
    - 50 条 `state.recent_answers` 非空
    - 52 条非终止 record 中，有 24 条的 `state.recent_answers` 已经包含 gold `Answer`
  - 也就是说，接近一半的非终止训练样本，在当前 state 中已经显式带有正确选项字母
- 影响判断：
  - 中高风险
  - 这不一定是“错误”，因为这些答案确实是 agent 在当时已经观察到的
  - 但如果你的目标是训练一个能在早期状态也有规划能力的世界模型，这会显著拉高后期样本的 shortcut 比例
- 建议人工再排查：
  - 分开评估“早期 step”和“晚期 step”的预测误差
  - 如果模型主要在晚期 step 上表现好，考虑：
    - 降低 answer-bearing state 的权重
    - 或按 step index 分层采样
    - 或分别训练 early-step / late-step heads

## R6. `done` 完全与 `TerminatorAgent` 绑定，done head 很容易偷懒

- 排查目的：
  - 判断 `done` 头是否真正学会“何时终止”
- 潜在风险：
  - 如果所有 `done=True` 样本都对应 `TerminatorAgent`，模型可以只靠动作类型预测 done，而不理解状态是否已完成
- 相关代码：
  - 当前 adapter 的 action feature 明确包含 terminator 标记，见 `puppeteer/inference/policy/workflow_world_model.py:1135` 附近
- 排查结果：
  - 本批 23 条 `done=True` record 中，23/23 都是 `action.name == TerminatorAgent`
  - `done=True` 且 `action != TerminatorAgent` 的样本数为 0
- 影响判断：
  - 中高风险
  - 这是典型 shortcut
- 建议人工再排查：
  - 如果想训练可泛化的终止判断，建议增加：
    - “状态已满足终止条件但还没 terminate”的样本
    - 或移除/弱化动作中的 terminator 标志做 ablation

## R7. auxiliary target 绝大多数接近常量，尤其 `conflict_score` 和 `coverage_score`

- 排查目的：
  - 判断 auxiliary/uncertainty 头是否有足够有效监督
- 潜在风险：
  - 如果 target 过于单一，头部会退化成常量回归器
  - 指标可能不错，但几乎没有策略价值
- 代码位置：
  - `puppeteer/inference/policy/workflow_dataset_recorder.py:495-537`
  - `puppeteer/inference/policy/workflow_world_model.py:1151-1160`
- 排查结果：
  - 本批数据中：
    - `coverage_score = 1.0` 的记录：71/75
    - `conflict_score = 0.0` 的记录：71/75
    - `progress_score` 主要落在 `0.2 / 0.4 / 0.6`
    - `termination_readiness` 主要落在 `0.65 / 1.0`
  - 当前 `uncertainty` 头直接监督 `conflict_score`，因此这批数据上的 uncertainty 标签几乎是常量
- 影响判断：
  - 中风险
  - 不是“字段错了”，而是“有效信息密度很低”
- 建议人工再排查：
  - 分任务类型统计这些 auxiliary label 的分布
  - 如果长期高度单一，建议重新设计 dense target，而不是继续堆 loss

## R8. reward/value 分布偏正，路径成功率很高

- 排查目的：
  - 判断训练数据是否出现明显的标签分布偏斜
- 潜在风险：
  - 世界模型会更容易学到“整体偏乐观”的 dynamics/value
  - 在失败/冲突状态上的泛化会更弱
- 排查结果：
  - 本批 75 条 record：
    - `outcome.reward > 0`：65
    - `outcome.reward < 0`：10
  - 23 条终止 path 中：
    - 正终止：20
    - 负终止：3
- 影响判断：
  - 中风险
  - 当前数据更像“成功轨迹为主，少量失败轨迹补充”
- 建议人工再排查：
  - 检查训练后模型在负样本上的误差是否显著更大
  - 必要时增加失败样本、对负样本重加权、或按 terminal reward 做重采样

## R9. `state.path_id` 在 split path 上存在不一致，但当前 adapter 不读取它

- 排查目的：
  - 判断 split path 是否把错误的 path 身份写进当前 state
- 潜在风险：
  - 如果后续有人直接用原始 JSONL 而不是当前 adapter，就可能把错误 path_id 当特征
- 相关代码：

  - `puppeteer/inference/reasoning/path.py:125`
  - 当前 adapter 主构图逻辑见 `puppeteer/inference/policy/workflow_world_model.py:982-1160`

  - 具体表现：
    - 所有 path 的首 step 都是 `state.path_id = -1`

## 6. 当前对训练影响最大的结论

如果只从“对当前 world model 训练效果的实际影响”排序，我建议优先关注：

1. `step_data` / `reasoning_results` / `tool_results` 的硬截断
2. `valid_action_mask` 恒定且包含未出现 agent
3. action 词表混合了 agent 名与动作名
4. scheduler JSON 解析失败导致采样路径带 fallback 噪声
5. later-step state 中频繁直接出现答案字母
6. `done` 与 `TerminatorAgent` 完全绑定

其中：

- 1/2/3 是结构性问题，优先级最高
- 4 是采样过程噪声问题，会污染 path 分布
- 5/6 是 shortcut 问题，会让模型“看起来能学会”，但学到的能力不一定是你真正想要的

## 7. 建议的人工复核清单

建议人工优先再做以下复核：

1. 从这 10 个 episode 中各抽 1 条最长的 search/reasoning step，人工判断前 300 字是否足以保留真正关键证据
2. 明确 valid-action 头到底想预测 agent 还是 action verb，并据此重构词表和标签
3. 抽查 5 次 scheduler fallback 的原始回复，确认 fallback 选中的 agent 是否真的符合 intended JSON
4. 把训练集按 `t=0/1` 与 `t>=2` 分开做评估，看模型是否只是在晚期 answer-bearing state 上表现好
5. 对 `done` 头做一个 ablation：移除 terminator 标志后看性能跌幅，估计 shortcut 程度
6. 看负样本与失败路径是否太少，必要时专门补失败 episode
7. 给采样主流程加“run 完成标记”或“dataset 落盘成功标记”，便于排除未完成 run

## 8. 最终判断

这批数据不是“坏数据”，核心 transition 语义整体是成立的。

但如果直接拿它训练 world model，模型很可能会同时学到三类东西：

1. 一部分真实的 workflow dynamics
2. 一部分由文本硬截断带来的不完整状态表示
3. 一部分由常量标签、答案暴露、Terminator shortcut 和 scheduler fallback 带来的捷径/噪声

因此更准确的结论是：

- 当前数据“可用，但不干净”
- 如果只做概念验证，可以先训
- 如果目标是做真正可用的世界模型，建议至少优先修：
  - 文本摘要策略
  - valid action 标签
  - action 语义词表
  - scheduler fallback 可观测性
