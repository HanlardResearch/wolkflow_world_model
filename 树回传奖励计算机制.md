# 树回传奖励计算机制

本文结合一次真实采样日志，解释 `REINFORCE_continuous.py` 里“调用树回传奖励”的实际逻辑、数据流和落盘方式。

## 1. 先说结论

这段代码里同时存在两套奖励信号：

1. `trajectory_step["reward"]`
   这是轨迹自身的逐步 reward。前期主要是按 step 深度和 agent 类型给一个负成本，任务结束时再结合 token cost 和终局结果修正最后一步 reward。
2. `world_model_*`
   这是“树回传”算出来的 world model 监督信号。它不是直接用上面的逐步 reward 反推，而是把多条已完成 path 组成一棵前缀树，再用每个前缀节点下所有叶子的终局 reward 均值做 credit assignment。

最关键的一点是：

- 本次日志里树回传真正使用的叶子标签是 `transition rewards: [1, -1, 1, -1]`
- 不是日志中每条 path 最后打印出来的 `Last Reward`

也就是说：

- `Last Reward` 更像“轨迹/策略训练侧”的最终一步 reward
- `world_model_step_credit / world_model_reward / world_model_mc_return` 等字段，来自“前缀树上的均值回传”

## 2. 代码入口在哪里

核心代码在以下几个位置：

- `puppeteer/inference/policy/REINFORCE_continuous.py:1492`
  `finalize_task_batch`
- `puppeteer/inference/policy/REINFORCE_continuous.py:1538`
  `_prepare_trajectory_for_finalize`
- `puppeteer/inference/policy/REINFORCE_continuous.py:1589`
  `_build_world_model_tree_targets`
- `puppeteer/inference/policy/REINFORCE_continuous.py:1663`
  `_annotate_trajectory_with_world_model_targets`
- `puppeteer/inference/policy/workflow_dataset_recorder.py:106`
  `record_completed_trajectory` 中把这些字段写进 JSONL

对应代码流程可以概括为：

1. 调度时，把每一步 agent 选择写进 `current_trajectories`
2. 任务结束时，在 `finalize_task_batch` 里收集所有完成的 path
3. `_prepare_trajectory_for_finalize` 先把轨迹 step reward 用 token cost 放大，并补上最后一步
4. `_build_world_model_tree_targets` 用所有 path 的动作序列建前缀树，按叶子终局 reward 计算每个节点 value
5. `_annotate_trajectory_with_world_model_targets` 把树上算出的值写回每一步
6. `workflow_dataset_recorder.py` 优先把这些 `world_model_*` 字段落盘；如果没有，才退回普通 reward/return

## 4. 树回传到底怎么算

`_build_world_model_tree_targets` 的核心思想是：

### 4.1 节点 value

对任意前缀节点 `prefix`：

```text
node_value(prefix) = 该前缀下面所有叶子 terminal_reward 的平均值
```

例如本例中：

```text
V(<root>) = 0
V(Python) = 0
V(Python -> Reasoning) = 1
V(Python -> Question) = (-1 + 1 + -1) / 3 = -0.3333
V(Python -> Question -> Question) = 1
V(Python -> Question -> Python) = -1
V(Python -> Question -> Modifier) = -1
```

### 4.2 每一步的 step credit

对某一步动作：

```text
step_credit = action_value - parent_value
```

也就是：

```text
step_credit = V(走到这个子节点) - V(当前父节点)
```

含义不是“这一步即时拿到了多少 reward”，而是：

```text
这一步把你带到的那个分支，相比父节点平均水平，价值提升了多少
```

### 4.3 leave-one-out gap

代码里还会算：

```text
leave_one_out_gap = action_value - sibling_baseline
```

其中：

```text
sibling_baseline = 同一父节点下，其它兄弟分支 value 的均值
```

它表达的是：

```text
这一步选中的分支，比没选中的兄弟分支平均好多少
```

### 4.4 h2_return

配置文件 `puppeteer/config/policy.json:13` 中 `gamma = 0.99`。

代码里还会计算一个短窗口回报：

```text
h2_return_t = step_credit_t + gamma * step_credit_{t+1}
```

这里只看 2 步 horizon，不是整条轨迹的完整折扣 return。

## 5. 用这次样本手工算一遍

下面把 4 条 path 的关键树回传值直接列出来。

记号简写：

```text
P = PythonAgent_gpt4o
R = ReasoningAgent_gpt4o
T = TerminatorAgent
Q = QuestionAgent_gpt4o
M = Modifier_gpt4o
C = CriticAgent_gpt4o
```

## 5.1 path_0

```text
path_0 = P -> R -> T
leaf reward = +1
```

逐步计算：

```text
step 0: P
parent = <root>, V(parent)=0
action = P, V(action)=0
step_credit = 0 - 0 = 0

step 1: R
parent = P, V(parent)=0
action = P->R, V(action)=1
step_credit = 1 - 0 = +1

step 2: T
parent = P->R, V(parent)=1
action = P->R->T, V(action)=1
step_credit = 1 - 1 = 0
```

解释：

- 第一步 `P` 没有拉开差距，因为所有 path 都先走了 `P`
- 第二步选 `R` 非常关键，因为它把 path 带到了一个纯正样本分支，节点值从 `0` 提高到 `1`
- 最后 `T` 只是把成功分支收尾，没有继续提升 value

## 5.2 path_1

```text
path_1 = P -> Q -> P -> Q -> P
leaf reward = -1
```

逐步计算：

```text
step 0: P
step_credit = 0

step 1: Q
parent = P, V(parent)=0
action = P->Q, V(action)=-0.3333
step_credit = -0.3333

step 2: P
parent = P->Q, V(parent)=-0.3333
action = P->Q->P, V(action)=-1
step_credit = -0.6667

step 3: Q
parent = -1, action = -1
step_credit = 0

step 4: P
parent = -1, action = -1
step_credit = 0
```

解释：

- 走到 `Q` 这个大分支时，平均值已经从 `0` 掉到 `-0.3333`
- 再走到 `P->Q->P` 这条子分支时，值进一步掉到 `-1`
- 后面继续在这个负分支内部展开，不再改变节点均值，所以 `step_credit` 变成 0

## 5.3 path_2

```text
path_2 = P -> Q -> Q -> R -> C
leaf reward = +1
```

逐步计算：

```text
step 0: P
step_credit = 0

step 1: Q
parent = P, V(parent)=0
action = P->Q, V(action)=-0.3333
step_credit = -0.3333

step 2: Q
parent = P->Q, V(parent)=-0.3333
action = P->Q->Q, V(action)=1
step_credit = +1.3333

step 3: R
parent = 1, action = 1
step_credit = 0

step 4: C
parent = 1, action = 1
step_credit = 0
```

这是本例里最值得注意的一条：

- 虽然它在 step 1 先进入了整体偏负的 `Q` 大分支
- 但在 step 2 又从 `Q` 的三个子分支里走到了唯一的正分支 `Q->Q`
- 所以这一跳的 `step_credit = 1 - (-0.3333) = +1.3333`

这就是“树回传”的价值：

- 它能把“在一个大类坏分支里，继续挑出了一个好子分支”的收益单独识别出来

## 5.4 path_3

```text
path_3 = P -> Q -> M -> M -> P
leaf reward = -1
```

逐步计算：

```text
step 0: P
step_credit = 0

step 1: Q
step_credit = -0.3333

step 2: M
parent = P->Q, V(parent)=-0.3333
action = P->Q->M, V(action)=-1
step_credit = -0.6667

step 3: M
step_credit = 0

step 4: P
step_credit = 0
```

它和 path_1 的逻辑类似：

- 先进入均值偏负的 `Q`
- 再进入一个确定负回报的子分支 `M`
- 一旦进入纯负子树，后续动作不再改变 prefix value，credit 就归零

## 6. leave-one-out gap 怎么理解

这个量特别适合看“兄弟分支之间的相对优劣”。

### 6.1 在 `P` 节点处分叉

`P` 的两个孩子是：

```text
P -> R    value = +1
P -> Q    value = -0.3333
```

所以：

```text
对于 path_0 的 step1 选 R:
leave_one_out_gap = 1 - (-0.3333) = +1.3333

对于 path_1/2/3 的 step1 选 Q:
leave_one_out_gap = -0.3333 - 1 = -1.3333
```

这很好理解：

- 选 `R` 明显优于另一个兄弟 `Q`
- 选 `Q` 明显差于另一个兄弟 `R`

### 6.2 在 `P -> Q` 节点处分叉

`P -> Q` 的三个孩子是：

```text
P -> Q -> P    value = -1
P -> Q -> Q    value = +1
P -> Q -> M    value = -1
```

所以：

```text
path_2 在 step2 选 Q:
sibling_baseline = (-1 + -1) / 2 = -1
leave_one_out_gap = 1 - (-1) = +2

path_1 在 step2 选 P:
sibling_baseline = (1 + -1) / 2 = 0
leave_one_out_gap = -1 - 0 = -1

path_3 在 step2 选 M:
sibling_baseline = (-1 + 1) / 2 = 0
leave_one_out_gap = -1 - 0 = -1
```

这说明：

- `path_2` 的第二步是一次非常强的好选择
- `path_1` 和 `path_3` 的第二步，则都把 path 带到了明显差的子分支

## 7. 为什么日志里的 `Last Reward` 不能直接当树叶 value

`train.log` 中同一批 path 还能看到：

```text
Last Reward:  1.0368483066558838
Last Reward: -1.0499999523162842
Last Reward:  1.0499999523162842
Last Reward: -1.0499999523162842
```

它们来自 `_prepare_trajectory_for_finalize`，是把：

- 终局对错 reward
- step cost
- 结束动作的额外 step reward

混合后写进轨迹最后一步的值。

但 `_build_world_model_tree_targets` 用的不是它，而是：

```python
terminal_reward = float(item["terminal_reward"])
```

其中 `terminal_reward` 在 `finalize_task_batch` 中来自：

```python
float(transition.get("reward", 0.0))
```

也就是这次日志里的：

```text
[1, -1, 1, -1]
```

所以你可以把它理解成：

- 轨迹 reward：更偏向“策略训练/成本感知”
- 树回传 reward：更偏向“结构 credit assignment / world model 监督”

## 8. 最终写进 world model 数据集的是什么

`workflow_dataset_recorder.py:137` 开始会组装每一步 record。

如果某一步已经被 `_annotate_trajectory_with_world_model_targets` 写入了树回传字段，那么 JSONL 中会优先使用这些值：

### 8.1 outcome

```text
outcome.reward <- world_model_reward
```

在本实现里，`world_model_reward` 实际上就是该动作子节点的 `action_value`。

### 8.2 returns

```text
returns.mc_return <- world_model_mc_return
returns.h2_return <- world_model_h2_return
```

本实现里：

- `world_model_mc_return = action_value`
- `world_model_h2_return = step_credit_t + gamma * step_credit_{t+1}`

### 8.3 credit_targets

```text
credit_targets.step_credit <- world_model_step_credit
credit_targets.leave_one_out_gap <- world_model_leave_one_out_gap
```

### 8.4 metadata

还会带上：

```text
metadata.tree_parent_value
metadata.tree_action_value
metadata.tree_sibling_baseline
metadata.tree_descendant_leaves
metadata.tree_leaf_reward
```

这些字段对分析 credit 分配非常有用。

## 10. 一句话总结

这段“树回传奖励”本质上做的是：

```text
把多条完成轨迹组成一棵动作前缀树
-> 用叶子终局 reward 的平均值定义每个前缀节点 value
-> 用 子节点 value - 父节点 value 作为 step_credit
-> 再把这些树上的 credit 写回每一步，作为 world model 训练目标
```

所以它不是普通 REINFORCE 那种“沿时间维折扣回报”，而是“沿动作前缀树做 credit assignment”。
